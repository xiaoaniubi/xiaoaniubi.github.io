数据结构

在大部分 Linux 操作系统种，内存的初始化一般都在 bootloader 中完成，然后在将内存大小传给 Linux 内核，因此从 Linux 内核的角度来看，内存其实就是一段物理内存空间。

在 Linux 内核中，和内存硬件物理特性相关的一些数据结构主要集中在 MMU（页表、高速缓存/TLB 操作等）中。所以 Linux 内核中内存管理的数据结构基本都是软件层面的概念，比如:

  • mm（struct mm_struct）
  • VMA（struct vm_area_struct）
  • 内存管理区（struct zone）
  • 页面（struct page）
  • 内存结点（struct pglist_data）

物理内存数据结构

与物理内存相关的数据结构有：

  • 内存结点（pglist_data）
  • 内存管理区（zone）
  • 物理页面（page）
  • mem_map[] 数组（用于存放 page 数据结构）
  • 页表项（PTE）
  • 页帧号（PFN）
  • 物理地址

其中 pglist_data 数据结构用来描述一个内存结点的所有资源。在 UMA 架构中只有一个内存节点

mm/memblock.c

include/linux/mmzone.h

而在 NUMA 架构中，系统的内存由一个 pglist_data * 的指针数据 node_data[] 来管理

arch/arm64/include/asm/mmzone.h

arch/arm64/mm/numa.c

Linux 内核为什么使用内存管理区来划分物理内存呢？

  • 32 位处理器由于地址数据线位宽的限制，最多支持 4GB 的物理内存以及虚拟地址空间。当前，如果开了 LPAE 可以支持更大的物理内存。在 4GB 的虚拟地址空间中，通常内核空间只有 1GB 的虚拟地址空间，所以 4GB 的物理内存是不能全部一一线性映射到内核虚拟地址空间的。同时不可能把 4GB 全部的虚拟地址空间都分给内核，所以用户和内核虚拟地址空间就有一个比例，此比例可以配置为 1:3、2:2、3:1，一般我们配置为 VMSPLIT_3G，即用户空间 3G，内核空间 1G。

    1 GB 内核空间最多只能线性映射 1GB 的物理内存，那剩下的物理内存就无法访问了，所以 Linux 把物理内存分成两部分，一部分为线性映射区，用管理区管理的话就是 ZONE_NORMAL，剩下部分就是高端内存，使用 ZONE_HIGHMEM 管理区来管理。线性映射区大小一般为 760MB，也就是说 0 ~ 760M 的物理内存可以线性映射到内核虚拟空间的线性映射区（3G ~ 3G + 760M）。剩下的物理内存都需要从 264M 的虚拟内存空间中拿出一部分来动态映射访问了。

  • 内存管理的划分和架构有关。比如在 x86 架构中，ISA 设备只能访问物理内存的前 16 MB，所以在 x86 中有一个名为 ZONE_DMA 管理区。在 x86_64 架构中，因为有足够大的空间可以全部线性映射物理内存，所以就不需要 ZONE_HIGHMEM 管理区了。

在 Linux 中，不管是什么架构，常见的所有管理区有：

  • ZONE_DMA：用于 x86 下 ISA 设备的 DMA 操作，范围为 0 ~ 16MB
  • ZONE_DMA32：在 64 系统中，外设 DMA 的寻址能力增强，因此又加入了这个管理区，寻址空间范围为 16MB 到 4GB
  • ZONE_NORMAL：大于 4GB 的物理内存，用于线性映射物理内存。如果系统内存小于 4GB 就没有这个管理区了
  • ZONE_HIGHMEM：32 位系统中管理高端内存，这些高端内存不能线性映射到物理内存，是页表映射的。64 位系统是没有这个管理区的。

处理器按照数据位宽寻址，也就是字，但是处理器在处理物理内存时不是按照字来分配的，现在的处理器都是采用分页机制来管理内存。CPU 中有一个 MMU 硬件单元，它会处理虚拟内存到物理内存的映射关系，也就是页表翻译工作。从 CPU 角度看，管理物理内存的最小单位为页。Linux 中使用一个 page 数据结构来表示一个页面。一个页面的大小默认为 4KB，但有的架构也可配置成 16KB、64KB 的页面。

Linux 内核为每个物理页面分配了一个 page 数据结构，如果采用 FLATMEM 内存模型，就是用一个 mem_map[] 静态数组才存放这些 page 数据结构，数组中每一个成员都对应一个物理页面。
