Dynamic DMA mapping using the generic device

这篇文档描述了 DMA API。有关 API 更简洁的介绍（实际例子），可以看 Documentation/DMA-API-HOWTO.txt

这篇文档分为两部分，Part I 描述了最基本的 API，Part II 描述了支持非一致性内存机制的扩展。除非你的驱动必须支持非一致性平台（这个通常只有旧的平台才有），否则你应该使用最基本的 API

DMA API

想要使用 dma API，你必须包含 linux/dma-mapping.h, 这个头文件介绍了 dma 物理地址类型 dma_addr_t 和一些接口。

dma_addr_t 类型可以表示所有平台的有效的 DMA 地址，这个地址可以直接作为外设的 DMA 源地址或 DMA 目标地址。cpu 不能直接使用 dma_addr_t 类型的地址，因为 dma_addr_t 地址空间可能需要转换位物理地址才能被 cpu 看到。

使用大的 DMA-coherent buffers

Consistent memory 的意思是 cpu 或 device 写数据后能理解被 device 或 cpu 看到，根本不用担心 cache 的一致性问题。（然而，你可以需要确保在 device 读内存之前要把 cpu 的 buffer 刷到内存中）。

调用 dma_alloc_coherent 成功后会返回 cpu 侧的虚拟地址，失败后会返回 NULL。

同时也会返回 DMA 的总线地址到 dma_handle。

并且调用 dma_alloc_coherent 返回的地址空间最小为 PAGE_SIZE，所以如果你想分配小于 PAGE_SIZE 的地址空间的话，调用 dma_pool 是你最好的选择。

只有 dma_alloc_coherent 允许调用者指定 flag 参数为 GFP_ （参考 kmalloc()，当然可以选择忽略这个标志，如 GFP_DMA）。

释放一块之前分配的一致性内存区域，参数 dev、size、dma_handle 必须和之前传给 dma_alloc_coherent() 的一样。cpu_addr 也必须是调用 dma_alloc_coherent() 的返回值。

需要注意：这个函数不能在中断上下文中调用。

使用小的 DMA-coherent buffers

想要使用 dma_pool，必须 #include <linux/dmapool.h>

许多驱动需要很多比较小的 DMA 一致性的内存区域，比如 I/O buffers 的 DMA 描述符。与其使用 dma_alloc_coherent() 分配以 page 为单位的内存，还不如使用 DMA pools。dma_pool 就像 struct kmem_cache 一样，唯一不同的是 dma_pool 使用 DMA 一致性分配器，而 kmem_cache 使用 __get_free_pages()。同样，他们都有硬件对齐的约束。

dma_pool_create() 初始化一个 DMA 一致性 buffer 的池子以供设备使用。它可能会睡眠，所以只能进程上下文中调用。

参数 name 是为了诊断（像 struct kmem_cache name）；参数 dev 和 size 参数和传给 dma_alloc_coherent() 的一样。参数 align 是为了硬件对齐需求（单位为 bytes，必须是 2 的 指数幂）。如果你的设备没有边界限制，传 0 就可以；如果传 4096，说明这次的内存分配不能超过 4KB 的边界。

dma_pool_zalloc 封装了 dma_pool_alloc() ，如果分配成功，返回的内存区域内数据全为 0

调用 dma_pool_alloc 可以从前面创建的池子中返回内存对齐的区间。如果传 GFP_ATOMIC 不会阻塞，传 GFP_KERNEL 可以阻塞（但不能在中断上下文中调用，并且也不能持有 SMP 锁）。跟 dma_alloc_coherent() 一样，也是返回两个值：一个是 cpu 测可访问的虚拟地址，另一个是 device 测可访问的 DMA 总线地址。

调用 dma_pool_free 将释放内存到池子里。这个 pool 就是之前传给 dma_pool_alloc() 的参数； cpu 虚拟地址（vaddr）和 DMA 地址是之前分配是返回的。

调用 dma_pool_destroy() 将释放 pool 资源。它必须在进程上下文中调用（因为可能会睡眠）。同时必须保证在销毁 pool 之前先释放所有的内存到 pool。

注意：

并不是所有的内存区域都可以使用这个 API 来映射的。虚拟内存连续，物理内存不一定连续。如果这个 API 不支持 scatter/gather 能力，那使用不连续的物理地址会映射失败。原因是，通过这个 API 映射的内存必须保证是物理连续的（像 kmalloc）。

此外，DMA 地址也必须在 device 的 dma_mask 范围内（dma_mask 是 device 可以访问的地址空间的位掩码，如果 DMA 地址和 dma_mask 相与等于 DMA 地址，那说明 device 可以操作这个 DMA 地址）。为了保证 kmalloc 分配的内存在 dma_mask 范围内，驱动可以指定各种平台的标志来限制 DMA 地址的范围（如，x86 平台的 GFP_DMA 保证在可用 DMA 地址的前 16MB 内，这是ISA设备所要求的 ）。

还请注意，如果平台具有 IOMMU (将 I/O dma 地址映射到物理内存地址的一个硬件)，则上述对物理连续性和 dma 掩码的限制可能不适用。然而，为了便于移植，设备驱动程序编写者可能不假设存在这样一个 IOMMU。

警告：

内存一致性以称为缓存线宽度的粒度进行操作。为了使这个API映射的内存能够正确运行，映射的区域必须精确地开始于缓存行边界，并精确地结束于缓存行边界（防止两个单独映射的区域共享一个缓存行）。由于在编译时可能不知道缓存行大小，因此API不会强制执行此要求。因此，建议那些在运行时不特别注意确定缓存行大小的驱动程序开发者们只映射在页面边界(也保证是缓存行边界)开始和结束的虚拟区域。

DMA_TO_DEVICE 同步必须在软件最后一次修改内存区域之后，在它被移交给设备之前完成。一旦使用了这个原语，该原语覆盖的内存应该被设备视为只读。如果设备在任何点写内存，那 DMA 应该是双向。

DMA_FROM_DEVICE 同步必须在被设备更改的数据之前完成。

DMA_BIDIRECTIONAL 需要特殊的处理：这意味着驱动程序不确定内存是否会在被移交给设备之前修改，也不确定设备是否也会修改它。因此，你必须始终同步 bidirectional 内存两次：

  • 一次是在内存移交给设备之前(确保所有最新数据都从 cache 刷新到内存)。
  • 一次是在数据被设备使用后并且能被 cpu 可以访问之前(确保任何处理器缓存行都更新了设备可能更改的最新数据)。

解除之前映射的区域。所有的参数必须和之前 map 的一样。

用于 MMIO 资源 mapping 和 unmapping 的 API。对于其他 mappping API，所有注意事项和警告也同样使用。这些 API 应该只用于映射设备 MMIO 资源，不允许映射 RAM。

在某些情况下，dma_map_single(), dma_map_page() 和dma_map_resource()会映射失败。驱动可以通过调用 dma_mapping_error() 来检查返回的 DMA 地址。返回非 0，表示不能创建映射，驱动应该采取一些行动处理这种情况（例如减少当前 DMA 映射的使用或者等一会儿再重试)。

返回值: 映射的 DMA 地址段的数量(如果 scatter/gather 列表的一些元素在物理上或实际上是相邻的，并且 IOMMU 将它们映射为单个条目，则可能比传入的 短，也就是说两个 scatterlist 的头尾连续，就可以当成一块来处理)。

请注意，如果已映射过一次，则不能再次映射。映射过程中是可以破坏 sg 中的信息的。

高级的 DMA 使用

警告: 这部分 DMA API 不应该在大多数情况下使用，因为它们满足不属于常规驱动程序的不太可能的极端情况。

如果你不知道处理器和 I/O 设备之间的缓存一致性的工作原理，那你根本就不应该使用这些 API。

与 dma alloc coherent() 相同，除了在 attrs 参数中传递 DMA_ATTR_NON_CONSISTENT 标志时，平台将根据其认为合适的情况选择返回一致或非一致内存。通过使用这个 API，你可以向平台保证，如果驱动程序选择返回非一致的内存，那么你在驱动程序中要确保该内存的所有正确和必要的同步点。

注意:在平台可以返回一致性内存的地方，它将保证同步点成为 nops。

警告:处理非一致内存确实很麻烦。只有当你确实知道驱动程序需要在一种罕见的(通常是非 PCI)架构上工作，而这种架构根本无法产生一致性的内存时，你才应该使用这个 API。

释放由 dma_alloc_attrs() 分配的内存。所有的公共的参数必须和 dma_free_coherent 一样，而 attrs 参数必须和 dma_alloc_attrs() 一样。

返回处理器缓存对齐。这是在映射内存或执行部分刷新时必须观察到的绝对最小对齐和宽度。

注意：

此 API 可能返回一个比实际缓存行大的数字，但它将返回一个或多个缓存行宽度。它也将始终是2的幂次方，以便于对齐。

对使用 DMA_ATTR_NON_CONSISTENT 标志，由 dma alloc attrs() 分配的内存进行部分同步，从虚拟地址 vaddr 开始，大小为 size。同样，在执行此操作时，你必须观察缓存行的边界。

声明由 dma alloc coherent() 在为该设备请求一致性内存时分配的内存区域。

phys_addr 是当前分配的物理内存（它将 ioremap 为 cpu 可以访问的虚拟内存区域）

device_addr 是设备需要被编程来实际寻址该内存的 DMA 地址(这将作为 dma_alloc_coherent() 的 dma_addr_t 参数)。

size 是那个区域的大小（必须是 PAGE_SIZE 的倍数）

flags 可以组合在一起：

  • DMA_MEMORY_EXCLUSIVE - 只从声明的区域分配内存。当在声明的区域中分配不到内存时，不允许 dma alloc coherent() 分配其他系统内存。

作为平台的简化，每个设备只能声明一个这样的内存区域。

出于效率的考虑，大多数平台选择仅在页面的粒度上跟踪声明的区域。对于较小的分配，您应该使用 dma_pool() 。

移除之前声明的内存区域。此 API 对该区域不执行是否在使用中的检查，并将无条件返回删除所有所需结构。确定内存区域目前是否被使用是驱动工程师的职责。

这用于占用声明空间的特定区域(dma alloc coherent()将分发找到的第一个空闲区域)。

device_addr 是像那个区域请求的 DMA 地址.

size 是那个区域的大小（必须是 PAGE_SIZE 的倍数）

返回值要么是指向 cpu 虚拟地址的指针，要么是个错误（通过 PTR_ERR()），如果该区域的任何部分被占用的话。

调试使用 DMA-API 的驱动程序

如上所述的 DMA-API 有一些限制。例如，释放 DMA 地址时必须使用相同大小的对应函数。随着硬件 IOMMUs 的出现，对于驱动程序来说，不违反这些约束变得越来越重要。在最坏的情况下，这种违反可能导致数据损坏，甚至破坏文件系统。

为了调试驱动程序并发现 DMA-API 使用中的 bug，可以将检查代码编译到内核中，它将告诉开发人员这些违规行为。如果你的体系结构支持它，您可以在内核配置中选择 ”Enable
debugging of DMA-API usage“ 选项。启用该选项会对性能产生影响，不要在生产内核中启用它。

如果开启了上面的配置，内核将会包含一些记录了为哪个设备分配了哪些 DMA内存的代码。如果这段代码检测到错误，它会在内核日志中打印一条带有一些详细信息的警告消息。一个示例警告消息可能如下所示：

驱动程序开发人员可以找到包括导致此警告的 DMA-API 调用的栈跟踪的驱动程序和设备。

默认情况下，只有第一个错误才会产生警告消息。所有其他错误将只静默计数。存在此限制是为了防止代码淹没内核日志。为了支持调试设备驱动程序，可以通过 debugfs 禁用此功能。详细信息请参阅下面的 debugfs 接口文档。

API 描述
dma-api/all_errors 该文件包含一个数值。如果这个值不等于零，调试代码将为它发现的每个错误打印一个警告到内核日志中。要小心使用这个选项，因为它很容易淹没您的日志
dma-api/disabled 如果调试代码被禁用，则此只读文件包含字符 ‘Y’。当内存耗尽或在引导时禁用它时，就会发生这种情况
dma-api/error_count 该文件是只读的，并显示发现的错误总数
dma-api/num_errors 这个文件中的数字显示在内核日志停止之前将有多少个警告被打印到内核日志中。这个数字在系统启动时初始化为 1，并通过写入这个文件来设置
dma-api/min_free_entries 可以读取这个只读文件以获得分配器所能看到的空闲 dma_debug_entries 的最小数量。如果该值降至零，代码将尝试增加 nr_total_entries 以进行补偿
dma-api/num_free_entries 分配器中空闲的 dma_debug_entries 的当前数目
dma-api/nr_total_entries 分配器中空闲和已使用的 dma_debug_entries 的总数
dma-api/driver-filter 您可以将驱动程序的名称写入此文件，以将调试输出限制为来自该特定驱动程序的请求。将一个空字符串写入该文件以禁用过滤器并再次查看所有错误

如果你将此代码编译到内核中，它将在默认情况下启用。如果你想在没有记录的情况下启动，你可以提供’dma_debug=off‘作为启动参数。这将关闭 DMA-API 调试。注意，你不能在运行时再次启用它。你必须重新启动。

如果你只想查看特定设备驱动程序的调试消息，可以指定 dma_debug_driver= 参数。这将在引导时启用驱动程序过滤器。之后，调试代码将只打印该驱动程序的错误。该过滤器以后可以使用 debugfs 禁用或更改。

当代码在运行时禁用自己，很可能是因为它耗尽了 dma_debug_entries ，无法按需分配更多的调试项。在启动时预分配 65536 个条目-如果这个值太低，您可以使用 ‘dma_debug_entries=‘ 来覆盖默认值。请注意，代码分批分配条目，因此预分配条目的确切数量可能大于实际请求的数量。每次动态分配与初始预分配相同数量的条目时，代码将打印到内核日志中。这是为了表明更大的预分配大小可能是合适的，或者一个驱动程序可能连续映射泄露。

dma-debug 接口 debug_dma_mapping_error() 用于调试无法在 dma_map_single() 和dma_map_page()接口返回的地址上检查 dma 映射错误的驱动程序。此接口清除了由 debug_dma_map_page() 设置的标志，以指示驱动程序已调用 dma_mapping_error()。当驱动程序执行 unmap 时，debug_dma_unmap() 检查标志，如果该标志仍然设置了，则打印警告消息，其中包括导致 unmap 的调用跟踪。可以从 dma_mapping_error() 例程调用此接口，以启用 dma 映射错误检查调试。
