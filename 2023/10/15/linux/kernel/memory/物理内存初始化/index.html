物理内存初始化

物理内存初始化

伙伴系统（buddy system）是操作系统常用的一种动态管理内存的方法。在用户提出申请时，分配一个大小合适的内存块给用户，并在用户释放后回收。在伙伴系统中，内存块的大小是 2 的 order 次幂个页面。内核中 order 的最大值用 MAX_ORDER 宏来表示。

我的 qemu 的 .config 文件中配置了 CONFIG_FORCE_MAX_ZONEORDER = 11，即MAX_ORDER = 11

也就是说把所有的空闲页面分组成 11 个内存块链表，每个内存块链表分别包括 1，2，4，8，16，…，1024 个页面。1024 个页面也就是 4MB 的连续物理内存。

物理内存在内核中用几个 zone 来管理空闲页块。zone 根据内核的配置来划分，如 qemu 中有 ZONE_DMA32 和 ZONE_NORMAL。

zone 结构中有一个 free_area 结构数组，大小为 MAX_ORDER。free_area 结构中还包含了一个 MIGRATE_TYPES 个链表。也就是 zone 中有 0 ~ (MAX_ORDER - 1) 个 free_area，每个 free_area 又有 MIGRATE_TYPES 个链表。

include/linux/mmzone.h

页面类型 描述
MIGRATE_UNMOVABLE 不可迁移的页面，一般是内核态分配的页面，若迁移可能造成崩溃
MIGRATE_MOVABLE 可迁移的页面，一般是用户态分配的页面
MIGRATE_RECLAIMABLE 可回收的页面
MIGRATE_PCPTYPES 每个 CPU 的本地缓存页面
MIGRATE_HIGHATOMIC 大块页面，为了减缓外部碎片化
MIGRATE_CMA 连续物理页面，如相机、不支持 IOMMU 的 外设DMA 等
MIGRATE_ISOLATE 分离页面，比如在页面迁移的时候先将页面放到分离链表中，表示页面处于分离状态，不能被释放。并且不能从这个类型中分配内存。

页面分配的状态可以通过 /proc/pagetypeinfo 查看

页块

内存管理中有个页块的概念，一个页块一般为 2^(MAX_ORDER - 1) 个页面。

include/linux/pageblock-flags.h

每个页块都有一个对应的 MIGRATE_TYPES 类型。zone 结构中有一个成员指针 pageblock_flags 指向用于存放所有页块 MIGRATE_TYPES 类型的内存空间。

pageblock_flags 指向的内存空间大小通过 usemap_size 来计算，每个页块用 4 位来存放类型。zone 的初始化函数 free_area_init_core 中会调用 setup_usemap 来计算和分配空间。

roundup 表示向上取整

  • 如果 zonesize 大小为 0 ~ 1023 个页面，都用 1 个 pageblock 表示
  • 如果 zonesize 大小为 1023 ~ 2047 个页面，都用 2 个 pageblock 表示

所以比如 zonesize = 1025，那就用两个页块表示，每个页块用 NR_PAGEBLOCK_BITS 位表示 MIGRATE_TYPES 类型，所以就有 2 x 4 = 8 位，然后再与 64 位 roundup，最终 usemapsize = 64，这个函数需要返回字节数，所以再除以 8 就可以了。

最后通过 memblock_alloc_node_nopanic 分配空间，并且让 zone->pageblock_flags 指向这个空间。

内核用两个函数来管理这些迁移类型，分别是 get_pageblock_migratetype 和 set_pageblock_migratetype

第 33 行，因为之前为 pageblock_flags 分配空间的时候返回的是 N 个字节，并且是向 64 位 word 取整，所以这里就是得到这个页块在第几个 word 中。

第 34 行，获取 word 内页块偏移

一个 word 可以容纳 16 个页块，比如这个 page 在第 17 个页块中，那么这个页块就在第 1 个 word（从 0 开始） 的第1 个页块中。因为 bitidx 从 1 开始算， 所以 bitidx = 1，word_bitidx = 0。

第 40 行，flags 也左移 60 位。

第 42 行，把旧的 word 值读出来。

第 44 行，生成新的 word 值并设置，将旧 word 中 bit[63:60] 清零，并且或上新的迁移类型标志 flags，bit[63:60] 刚好就是第 1 个 word 的第 1 个页块。

zone 初始化

初始化完页表之后，内核就可以对内存进行管理了。但是内核并不是统一对待这些页面，而是采用内存管理区来管理。

zone 经常被访问到，因此这个数据结构要求以 L1 高速缓存对齐。可以看到 ARM64 的 L1 缓存行大小为 2^6 = 64 字节。

include/linux/cache.h

arch/arm64/include/asm/cache.h

这里 ZONE_PADDING() 让 zone->lock 和 zone->lru_lock 两个被频繁访问的锁分别在两个不同的高速缓存行中，避免造成高速缓存伪共享问题。

可以看出 ZONE_PADDING 本身就是按 L1 高速缓存行对齐的。

一个内存节点最多有几个 zone，所以 zone 数据结构不需要像 page 结构一样关注其大小，所以 ZONE_PADDING 是为了性能而浪费一点空间。在内存管理开发过程中，zone->lock 和 zone->lru_lock 有时需要同时获取，所以让它们位于不同的高速缓存行是内核开发常用的一种优化技巧。在 Linux 5.0 内核后，zone->lru_lock 已经转移到了 pglist_data 数据结构中。

通常情况下，内核的 zone 分为 ZONE_DMA、ZONE_DMA32、ZONE_NORMAL、ZONE_HIGHMEM。

我的 qemu 中只配置了 CONFIG_ZONE_DMA32，所以只有 CONFIG_ZONE_DMA32 和 ZONE_NORMAL 两个 zone。

ZONE_DMA32 的大小定义在 zone_sizes_init 中，一般不会超过 4GB。

ZONE_DMA32 管理区的最大物理地址由 max_zone_dma_phys 得出。其中 GENMASK_ULL 表示第 32 ~ 64 位都是 1，其余位都是 0。有几种情况：

  • memblock 起始地址和结束地址都小于 4GB，就取 memblock 的结束地址
  • memblock 起始地址小于 4GB，结束地址大于 4GB，就取 4GB
  • memblock 起始地址和结束地址都大于 4GB，就取 offset + 4GB

从前两种情况下，ZONE_DMA32 管理区物理地址最大值不能超过 4GB。

而第三种情况，offset 为 0 ~ 1023 时，通过 PFN_DOWN 宏返回的也是和 4GB 对应的 pfn 一样。只有 offset 大于 1023 时，才会超过 4GB 对应的 pfn，而这是 memblock 起始地址已经 4TB 的位置了，非常大了，几乎有这种配置。

mm/memory.c

mem_map 大小是由函数 calc_memmap_size 计算的。

spanned_pages 表示 zone 中所有的物理页面数量，包括可能不存在的内存空洞，所以如果空洞太大，使用 mem_map 静态数组方式就会比较浪费内存空间。

所以代码中做了一点折中：

  • 如果空洞大小大于 present_pages 的 1/16，就使用 SPARSEMEM 内存模型，使用 present_pages 计算。
  • 如果空洞大小小于等于 present_pages 的 1/16，就使用 FLATMEM 内存模型，使用 spanned_pages 计算。
